# EduForge 项目总结与规划

> 更新时间：2026-02-15

---

## 一、产品定位

### 核心定位

**EduForge — 教育界的 OpenClaw。**

不是一个固定功能的教学工具，而是一个**开源、插件化、AI 原生的教育平台引擎**。和 OpenClaw 一样，核心引擎小而精，能力通过插件扩展，用户通过 AI 对话完成工作。

### 类比关系

```
OpenClaw                              EduForge
─────────────                         ─────────────
"每个人都有 AI 助手"          →      "每个师生都有 AI 教学助手"
核心 Agent 引擎               →      核心教学引擎
Skills 技能系统               →      教学插件系统
ClawhHub（技能市场）          →      EduForge Plugin Hub（插件市场）
对话式交互（主入口）          →      对话式交互（教师/学生各有 AI 助手）
Dashboard（辅助管理）         →      Dashboard（查看数据、手动管理）
多渠道（Telegram/飞书/...）   →      多端（Web/微信小程序/钉钉）
自托管 / 数据自控             →      学校私有部署 / 数据不出校
开源社区                      →      教师 + 开发者共建
```

### 一句话

**OpenClaw 让每个人都有 AI 助手，EduForge 让每个师生都有 AI 教学助手。**

### 核心理念

| 理念 | 说明 |
|------|------|
| **AI 原生** | AI 对话是主要交互方式，不是附加功能 |
| **插件化** | 核心只管基础设施，教学能力全部插件化 |
| **开源可控** | 学校可私有部署，数据不出校 |
| **社区驱动** | 插件市场 + 教师社区，共建教育生态 |

### 目标用户

| 用户 | 场景 |
|------|------|
| 学校 / 培训机构 | 统一部署，管理员配置 AI 和插件，多教师多班级 |
| 独立教师 | 个人使用，对话式出题批改，零门槛 |
| 学生 | AI 辅导、做作业、错题复习 |
| 家长 | 查看学情报告 |
| 开发者 | 开发教学插件，贡献到社区 |

### 竞品对比

| 维度 | EduForge | 猿题库/作业帮 | 学校自建系统 |
|------|----------|---------------|-------------|
| 交互方式 | AI 对话 + 表单 | 纯表单 | 纯表单 |
| 部署方式 | 私有化/云端 | 仅云端 | 私有化 |
| 扩展性 | 插件系统 | 封闭 | 定制开发 |
| 数据归属 | 用户自有 | 平台所有 | 用户自有 |
| AI 深度 | 核心能力 | 辅助功能 | 无 |
| 成本 | 低（开源） | 按人收费 | 高 |

## 二、AI 助手架构（待讨论）

### 2.1 核心体验

教师和学生各有 AI 助手，通过对话完成日常工作：

**教师助手示例：**
```
教师：帮我出10道七年级一元一次方程的题，难度中等
AI：✅ 已生成 10 道题并加入题库 [查看]

教师：用这些题给七年级一班布置作业，周五截止
AI：✅ 已布置，29 名学生将收到通知

教师：昨天的作业情况怎么样？
AI：已批改 27/29 份，班级平均 78 分。张三和李四在"移项"上错误较多，建议出针对性练习。
```

**学生助手示例：**
```
学生：这道方程题我不会做
AI：我来一步步引导你。首先，等式两边...（苏格拉底式引导，不直接给答案）

学生：帮我复习一下错题
AI：你最近在"二元一次方程组"错了3题，我们从第一题开始...
```

### 2.2 开放设计问题

以下问题需要讨论确定后再实现：

#### ❓ Q1: AI 配置层级 — 谁来配？怎么配？

**场景分析：**

| 部署场景 | 配置者 | 需求 |
|----------|--------|------|
| 学校部署 | 学校 IT 管理员 | 统一配 Key，控制用量预算，选择合规模型 |
| 培训机构 | 机构负责人 | 统一配，可能多校区共享 |
| 独立教师 | 教师自己 | 自己配 Key，个人使用 |
| SaaS 云端 | 平台方 | 平台提供默认 AI，按量计费 |

**可能的层级设计（类比 OpenClaw 的配置覆盖）：**

```
平台默认配置（SaaS 模式下平台方提供）
  └── 学校/机构级配置（管理员设置，覆盖默认）
        └── 教师个人配置（可选，覆盖学校级）
              └── 学生无需配置（继承学校/教师的配置）
```

**待决策：**
- 是否允许教师用自己的 Key 覆盖学校配置？
- 学生是否永远继承上层配置，还是高年级学生可以自带 Key？
- 用量配额如何分配？学校总额度 → 按班级/教师分？
- 不同任务（批改 vs 辅导 vs 出题）是否可以用不同模型？

#### ❓ Q2: AI 助手的能力边界 — 能干什么？

**确定的：**
- ✅ 调用已安装的插件功能（出题、布置作业、查学情...）
- ✅ 回答教学相关问题
- ✅ 根据上下文给出教学建议

**待讨论的：**

| 能力 | 说明 | 考虑 |
|------|------|------|
| **主动调用插件** | AI 根据对话意图自动选择并调用合适的插件 | OpenClaw 就是这么做的。技术上用 Tool Calling 实现。问题是权限控制——AI 调用是否需要二次确认？ |
| **查询插件市场** | "有没有适合英语听力训练的插件？" | 需要插件市场的元数据搜索能力。用户体验很好，但需要插件市场先建起来 |
| **安装插件** | "帮我安装那个听力训练插件" | 风险较高。需要审批流程——教师安装需管理员批准？还是教师有完全自主权？ |
| **配置插件** | "把 AI 批改的模型换成 GPT-4" | 涉及系统配置，需要权限控制 |
| **跨插件协作** | AI 综合多个插件的数据做分析 | 比如结合错题本 + 学情分析 + 知识图谱给出个性化建议 |

**类比 OpenClaw 的设计：**
- OpenClaw 的 Agent 可以自由调用所有 Skills → EduForge 的 AI 助手是否也应该自由调用所有插件？
- OpenClaw 对外部操作（发消息、发邮件）会谨慎 → EduForge 对哪些操作需要确认？（布置作业？发通知？）
- OpenClaw 的 ClawhHub 可以发现和安装 Skills → EduForge Plugin Hub 的安装权限如何设计？

#### ❓ Q3: 权限与安全

| 问题 | 选项 |
|------|------|
| 教师 AI 助手能看到所有班级数据吗？ | A. 只看自己的班级 B. 看全校（管理员可控） |
| 学生 AI 助手能看到其他同学的数据吗？ | 肯定不能，但 AI 需要严格限制 |
| AI 操作是否需要确认？ | A. 查询类自动执行，修改类需确认 B. 全部自动 C. 可配置 |
| AI 对话内容是否留存？ | 教学审计需要？隐私保护？ |

#### ❓ Q4: 技术实现路径

| 方案 | 说明 | 优劣 |
|------|------|------|
| **A. 自建 Agent** | 后端实现 Tool Calling + 对话管理 | 完全可控，但开发量大 |
| **B. 集成 OpenClaw** | EduForge 作为 OpenClaw 的一组 Skills | 复用 OpenClaw 的 Agent 能力，但架构耦合 |
| **C. 混合方案** | 核心 Agent 自建，复杂场景调用 OpenClaw | 灵活但复杂 |

## 四、差距分析

### 🔴 P0 — 核心缺失

| 缺失 | 影响 |
|------|------|
| **AI 对话助手** | 没有这个，就不是"教育界的 OpenClaw"，只是又一个表单工具 |
| **插件市场** | 没有生态就没有网络效应 |
| **微信登录/小程序** | 中国市场必备 |

### 🟡 P1 — 重要补充

| 缺失 | 影响 |
|------|------|
| 多学校/多租户 | 无法服务机构客户 |
| AI 自动出题 | 题库靠手动不可持续 |
| 知识点图谱 | 无法做个性化推荐 |
| 教材版本适配 | 人教版/北师大版等 |
| 移动端适配 | 手机用户体验 |

### 🟢 P2 — 体验增强

| 缺失 | 影响 |
|------|------|
| 家长端 | 付费入口 |
| 通知推送 | 及时性 |
| 数据导出 | 学校行政需求 |
| 离线支持 | 网络不稳定场景 |

## 五、工作规划

### Phase 0：AI 助手设计（1 周）

> 目标：确定 AI 助手的架构设计，这决定了整个产品的形态

- [ ] 讨论并确定 Q1-Q4 的设计决策
- [ ] 输出 AI 助手技术方案文档
- [ ] 确定 MVP 版本的能力范围

### Phase 1：MVP + AI 助手原型（2-3 周）

> 目标：能让真实教师通过对话完成基本教学工作

| 任务 | 说明 |
|------|------|
| AI 对话界面 | 前端聊天组件 + 后端 Agent（Tool Calling） |
| 基础 Tools | 出题、布置作业、查学情、查错题 |
| 微信登录 | 扫码或小程序 code 登录 |
| 题库扩充 | AI 批量生成到 500+ |
| 云端部署 | Docker 上线 |
| **找 3-5 个教师试用** | |

### Phase 2：产品打磨（3-4 周）

> 目标：根据反馈迭代，打磨 AI 助手体验

| 任务 | 说明 |
|------|------|
| 学生 AI 助手 | 苏格拉底式辅导，不给答案给引导 |
| 插件市场 v1 | 插件发现 + 安装（AI 可查询安装） |
| 知识点图谱 | 薄弱点分析 + 个性化推荐 |
| 多租户 | 学校/机构级隔离 |
| 家长端 | 微信小程序查看学情 |

### Phase 3：商业化（1-2 月）

| 方向 | 模式 |
|------|------|
| SaaS 订阅 | 基础免费 + 高级按年付费 |
| AI 用量计费 | 免费额度 + 超出按量 |
| 插件市场 | 开发者卖插件，平台抽成 |
| 私有部署服务 | 学校付费部署+维护 |

## 六、关键指标

### MVP 阶段目标

| 指标 | 目标值 |
|------|--------|
| 试用教师 | ≥ 5 人 |
| AI 对话满意度 | "有用"评价 ≥ 70% |
| 活跃学生 | ≥ 50 人 |
| 插件数 | ≥ 5 个 |

### 判断标准

- ✅ 教师通过对话完成 >50% 的日常操作（而非手动点按钮）
- ✅ 至少 1 个教师主动推荐给同事
- ❌ 1 个月内无自然增长 → 调整方向

## 七、风险

| 风险 | 应对 |
|------|------|
| AI 对话体验不够好 | 从最高频的 3 个场景打磨，不贪全 |
| 插件生态冷启动 | 先自建核心插件，社区后期跟进 |
| AI 成本控制 | 分级模型策略（简单任务用小模型） |
| 教师接受度 | 对话 + 传统 UI 并存，渐进式引导 |
